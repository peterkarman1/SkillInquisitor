# Epic 6/7 Design: Injection, Suppression, Structural Validation, and Metadata Integrity

**Date:** 2026-03-14
**Status:** Approved for planning
**Epic:** Epic 6 — Deterministic Checks: Injection & Suppression
**Epic:** Epic 7 — Deterministic Checks: Structural & Metadata

## Goal

Build the deterministic injection and structural-validation subsystem for SkillInquisitor so the scanner can identify known prompt-injection signatures, suppression directives, YAML frontmatter abuse, suspicious skill layouts, risky URLs, package poisoning, skill-name typosquatting, and text-density anomalies without overreaching into the semantic territory reserved for later ML and LLM layers.

After Epics 6 and 7, the project should have:
- structured `SKILL.md` frontmatter extraction and validation
- deterministic coverage for known injection and jailbreak signatures
- deterministic coverage for suppression and concealment directives with scoring-oriented metadata
- per-skill structural validation rather than path-only heuristics
- URL classification that works across original and derived segments
- package-registry override and typosquatting detection across common package ecosystems
- skill-name typosquatting detection based on parsed frontmatter
- robust text-density anomaly detection that does not punish normal UTF-8 content
- regression fixtures that lock down both malicious and safe lookalike cases in a way that remains compatible with later Epic 9 and Epic 10 findings

The key design choice is that Epics 6 and 7 should not be implemented as a pile of regexes on the current substrate. They need a small metadata-plumbing upgrade first so later rules are precise, testable, and reusable.

## Supersedes Current Architecture Text

This spec intentionally supersedes the current Epic 6 and Epic 7 sections in [architecture.md](/Users/peterkarman/git/SkillInquisitor/docs/requirements/architecture.md) in the following ways:

- frontmatter validation is no longer specified as a regex-oriented pass over raw markdown; it now requires structured `SKILL.md` frontmatter extraction plus token/event-level YAML inspection
- deterministic rule execution is no longer effectively limited to only `segment` and `artifact` scopes; `skill` scope is required for robust structure and name-validation rules
- file-size anomaly detection no longer uses a naive byte-to-visible-character ratio; it now uses corroborated display-density analysis to avoid false positives on normal UTF-8 content
- URL policy moves from the current flat `trusted_urls` posture toward a structured host/scheme/registry policy, while preserving backward compatibility

When implementation lands, `docs/requirements/architecture.md` should be updated to match this spec so the project returns to a single authoritative design.

## Approved Constraints

- Optimize for long-term architecture fidelity, not the shortest rule drop.
- Keep deterministic rules high-precision and complementary to Epic 9 ML rather than trying to own subtle semantic intent.
- Add only the minimum substrate required for robust deterministic behavior:
  - frontmatter extraction
  - skill-scope validation support
  - richer artifact metadata for binaries, executability, and byte size
  - richer fixture assertions for metadata-heavy findings
- Preserve the existing `Skill -> Artifact -> Segment` model and flat `artifact.segments` contract.
- Continue to treat derived segments as first-class scan targets.
- Keep deterministic findings transparent and individually testable.
- Prefer conservative severity for ambiguous structural findings and reserve stronger severities for truly dangerous evidence.
- Preserve compatibility with the existing `trusted_urls` config while expanding URL policy into a more durable shape.

## Non-Goals

Epics 6 and 7 do not implement:
- ML-based prompt injection classification
- semantic analysis of subtle or rephrased jailbreaks
- full AST parsing of install commands or package manifests across every ecosystem
- LLM validation of whether an injection-like phrase is actually operational
- generalized archive extraction or binary unpacking
- final scoring math for suppression amplification or cross-layer reinforcement

These epics should provide the deterministic evidence and metadata that later ML, LLM, and scoring work can consume.

## Approach

The approved approach is a **substrate-first deterministic expansion**.

Phase one upgrades the scanning substrate so deterministic rules can reason about:
- leading YAML frontmatter in `SKILL.md`
- parsed skill metadata such as `name`, `description`, and `disable-model-invocation`
- skill-scoped inventory and structure
- non-text files, executable files, and artifact byte size
- fixture-local config and metadata assertions

Phase two adds `injection.py` for Epic 6 and `structural.py` for Epic 7, both following the existing Epic 5 pattern:
- primary findings remain normal deterministic findings
- rules emit structured `details` and `action_flags` when needed
- later epics can reinforce, downgrade, or amplify those findings rather than re-invent them

This keeps responsibilities clean:
- `input.py` and `normalize.py` reveal the right metadata and segments
- rule modules decide what is suspicious
- `engine.py` orchestrates segment, artifact, and skill evaluation
- tests and fixtures remain the acceptance contract

## Required Substrate Changes

The current codebase is close, but not yet strong enough for a robust Epic 6/7 implementation.

### Frontmatter Extraction

`models.py` already declares `Artifact.frontmatter` and `SegmentType.FRONTMATTER_DESCRIPTION`, but the scanner does not currently populate them. Epics 6 and 7 should add structured extraction of leading YAML frontmatter only for `SKILL.md` artifacts whose content begins at byte 0 with `---`.

The frontmatter pipeline should:
- preserve the raw frontmatter block
- preserve source span information for exact finding anchoring
- parse exactly one top-level mapping
- continue scanning the artifact even if frontmatter is malformed
- emit a `FRONTMATTER_DESCRIPTION` segment when the parsed `description` field is present and string-like

Ownership contract:
- `normalize.py` is the single owner of frontmatter parsing and hydration of `Artifact.frontmatter*` fields during artifact normalization
- `pipeline.py` is the single owner of updating `Skill.name` after normalized artifacts are returned

Parse-result contract:
- if the frontmatter parses as one top-level mapping, `Artifact.frontmatter` stores that mapping
- if the YAML is valid but not a top-level mapping, `Artifact.frontmatter` becomes `{}` and `frontmatter_error` records `frontmatter must be a mapping`
- duplicate keys are preserved in `frontmatter_observations`; the parsed mapping may use last-key-wins semantics for storage, but `Skill.name` may only be updated when:
  - the parsed frontmatter is a mapping
  - `name` is a string
  - there is exactly one observed `name` field span
- otherwise `Skill.name` remains the provisional input-derived name

### Artifact Metadata and Skill Provenance

`input.py` currently skips undecodable files entirely. That prevents reliable D-14 and D-23 detection for executables, installers, archives, and opaque bundles. Epics 6 and 7 should preserve enough artifact metadata to classify:
- byte size
- text versus binary status
- encoding when known
- executable bit or shebang presence
- simple signature class for common binary formats

The `Skill` model should also distinguish between:
- declared skill directories
- synthetic directory scans
- synthetic file scans
- stdin-backed synthetic skills

That distinction matters because some structure checks should apply only to declared skills while direct-risk checks should still apply to synthetic targets.

### Skill-Scoped Deterministic Rules

`engine.py` should add native `skill` scope in addition to the current `segment` and `artifact` scopes. D-14 and skill-name typosquatting are naturally per-skill, and implementing them as artifact hacks would make the design brittle.

### Fixture-Harness Enhancements

`tests/conftest.py` should support two optional fixture features:
- assertion of selected metadata such as `action_flags` and specific `details` keys
- fixture-local config overrides so D-15 allowlist behavior and typosquatting policy can be verified without mutating global defaults

Malicious fixtures for these epics should default to deterministic-scoped expectations so later ML and LLM findings do not invalidate the deterministic regression suite.

## Architecture

Epics 6 and 7 should preserve the current pipeline shape while extending the deterministic layer's substrate and scope model.

### Core Ownership

The approved ownership is:

- `input.py`
  - artifact metadata capture
  - skill provenance classification
- `normalize.py`
  - frontmatter extraction
  - `FRONTMATTER_DESCRIPTION` segment creation
  - normalized content for derived frontmatter text
- `injection.py`
  - D-11, D-12, D-13 detection
- `structural.py`
  - D-14, D-15, D-20, skill-name typosquatting, D-23 detection
- `engine.py`
  - scope-aware deterministic orchestration and registration
- `pipeline.py`
  - unchanged high-level sequencing, but normalization should receive the active `ScanConfig`

### Deterministic Boundaries with Epic 9

Epic 6 must stay intentionally narrower than the future ML layer.

Deterministic should own:
- canonical jailbreak phrases
- exact role/channel delimiters
- explicit system-prompt mimicry markers
- explicit non-disclosure or confirmation-bypass language
- frontmatter structure abuse

Epic 9 ML should later own:
- paraphrased or socially engineered prompt injection
- subtle multi-sentence coercion
- borderline intent that requires semantic reading rather than exact signatures

If a deterministic rule needs broad synonym expansion or semantic interpretation to work, that is a sign the rule is crossing into ML territory.

## Model Changes

Epics 6 and 7 should extend the shared models conservatively.

### `Artifact`

The current `Artifact` model should gain these exact fields in addition to the existing ones:
- `byte_size: int = 0`
- `is_text: bool = True`
- `encoding: str | None = None`
- `is_executable: bool = False`
- `binary_signature: str | None = None`
- `frontmatter_raw: str | None = None`
- `frontmatter_location: Location | None = None`
- `frontmatter_error: str | None = None`
- `frontmatter_fields: dict[str, Location] = Field(default_factory=dict)`
- `frontmatter_observations: list[dict[str, object]] = Field(default_factory=list)`

`frontmatter` should remain the parsed value mapping for downstream consumers. `frontmatter_fields` is the exact-span contract for D-13 anchoring. `frontmatter_observations` is the persisted parser-event contract for non-field YAML evidence such as:
- duplicate keys
- anchors
- aliases
- merge keys
- explicit tags
- embedded document markers

Each observation should carry:
- `kind`
- optional `key`
- optional `value_preview`
- `location`

Non-text artifact contract:
- binary or otherwise non-text files should still be represented as `Artifact` instances
- for non-text artifacts:
  - `raw_content = ""`
  - `normalized_content = None`
  - `segments = []`
  - metadata fields such as `byte_size`, `is_text`, `is_executable`, and `binary_signature` remain populated
- `normalize.py` should skip segment extraction and text normalization when `is_text` is `False`

This keeps structural rules able to see the artifact while preserving the current text-segment model.

### `Skill`

The `Skill` model should keep `name`, but the value should be derived from parsed frontmatter when available instead of only from directory name. It should also gain:
- `scan_provenance: str = "declared_skill"`

Allowed values:
- `declared_skill`
- `synthetic_directory`
- `synthetic_file`
- `stdin`

### `Segment`

No major new segment shape is required beyond using the existing `FRONTMATTER_DESCRIPTION` type, but the shared location helpers should be corrected so child-segment matches can be anchored precisely in absolute file coordinates.

## Pipeline Sequencing

Epics 6 and 7 require one explicit sequencing contract because `Skill` objects are currently created before normalization.

The deterministic pipeline should run in this order:

1. `input.py` resolves raw skills and artifacts, captures filesystem metadata, and assigns provisional `Skill.name` plus `scan_provenance`
2. `normalize.py` processes each artifact using the active `ScanConfig`
3. if a normalized `SKILL.md` artifact exposes parsed frontmatter:
   - `pipeline.py` updates `Skill.name` from normalized artifact frontmatter when present and valid
4. deterministic rules run in scope order:
   - skill-scoped rules
   - artifact-scoped rules
   - segment-scoped rules
5. deterministic postprocessors run afterward

This sequencing makes frontmatter-derived skill metadata available before D-14 and D-20F execute.

## Epic 6 Rule Taxonomy

Epic 6 should use family-scoped sub-rule IDs rather than one broad detector per BRD requirement.

### D-11 Prompt Injection Patterns

- `D-11A` Instruction-hierarchy override
  - Exact phrases such as `ignore previous instructions`, `disregard system prompt`, `forget all prior instructions`
  - Severity: `HIGH`
- `D-11B` Role rebinding or persona takeover
  - Phrases such as `you are now`, `from now on act as`, `pretend to be`
  - Require nearby role or behavior context to avoid false positives
  - Severity: `MEDIUM`
- `D-11C` System-prompt extraction or disclosure request
  - Phrases such as `reveal the system prompt`, `print the hidden instructions above`
  - Severity: `HIGH`
- `D-11D` Role or channel delimiter injection
  - Tokens such as `<|system|>`, `<|im_start|>`, `[INST]`, `<<SYS>>`
  - Severity: `HIGH`
- `D-11E` System-prompt mimicry
  - Tokens such as `<system>`, `[SYSTEM]`, `### SYSTEM INSTRUCTIONS`
  - Severity: `MEDIUM`
- `D-11F` Canonical jailbreak signature
  - Tokens such as `DAN`, `developer mode`, and similarly well-known named jailbreak templates
  - Severity: `MEDIUM`

### D-12 Suppression Directives

Suppression findings should act primarily as severity amplifiers for later scoring rather than dominating verdicts on their own.

- `D-12A` Concealment or non-disclosure directive
  - `do not mention`, `without telling the user`, `do not report`
  - Severity: `MEDIUM`
- `D-12B` Silent execution directive
  - `silently`, `quietly`, `in the background` only when paired with an action
  - Severity: `LOW` or `MEDIUM`
- `D-12C` Output or reporting suppression
  - `do not show output`, `hide logs`, `suppress warnings`
  - Severity: `LOW` or `MEDIUM`
- `D-12D` Confirmation or audit bypass
  - `do not ask for approval`, `proceed without confirmation`
  - Severity: `MEDIUM`

Every D-12 finding should emit:
- `SUPPRESSION_PRESENT`
- one specific flag such as `SUPPRESS_DISCLOSURE`, `SUPPRESS_OUTPUT`, or `SUPPRESS_CONFIRMATION`

And should record structured metadata such as:
- `signal_family`
- `source_kind`
- `amplifier_eligible`
- `amplifier_scope`

### D-13 YAML Frontmatter Validation

D-13 should be structured and syntax-aware rather than a whole-file regex pass.

Implementation contract for D-13 parsing:
- extract frontmatter only from leading `SKILL.md` frontmatter beginning at byte 0
- perform token/event-level YAML inspection before plain object construction so duplicate keys, anchors, aliases, merge keys, explicit tags, and embedded document markers are still observable
- store parser errors in `Artifact.frontmatter_error` rather than failing the scan
- anchor structural findings to the best available token, key, or value span from `frontmatter_fields` or parser marks
- scan the emitted `FRONTMATTER_DESCRIPTION` segment separately for content-oriented D-11 and D-12 matches

- `D-13A` Unexpected frontmatter field
  - Default allowlist: `name`, `description`, `disable-model-invocation`
  - Severity: `LOW`
- `D-13B` Invalid frontmatter field type
  - Enforce string/bool types for known fields
  - Severity: `MEDIUM`
- `D-13C` Overlong description
  - Trigger at rendered description length greater than 500 characters
  - Severity: `LOW`
- `D-13D` YAML injection construct
  - Anchors, aliases, merge keys, duplicate keys, explicit tags, or embedded multi-document constructs
  - Severity: `HIGH`
- `D-13E` Action-oriented or injection-oriented description
  - Description text that contains D-11/D-12-style action directives
  - Severity: `MEDIUM` or `HIGH` depending on exact signal

Unknown fields should produce findings, not hard parse failures. Malformed frontmatter should not stop scanning of the rest of the artifact.

## Epic 7 Rule Taxonomy

Epic 7 should mix skill-scoped, artifact-scoped, and segment-scoped detection depending on the behavior being modeled.

### D-14 Skill Structure Validation

D-14 should be skill-scoped inventory validation.

Definition of a declared skill directory:
- a directory explicitly scanned that contains `SKILL.md`
- a directory discovered beneath a scan root via recursive `SKILL.md` discovery

Synthetic targets are:
- a directory scanned directly without a `SKILL.md`
- a single file scan
- stdin-backed content

Layout-conformance rules such as `D-14A`, `D-14B`, `D-14C`, and low-severity `D-14G` apply only to `declared_skill`.

There is no standalone "missing root `SKILL.md`" finding in the current input contract, because non-skill directories are represented as synthetic targets rather than malformed skills. That behavior may be revisited in a future input-model epic, but it is not part of Epic 7.

Default benign top-level allowlist for declared skills:
- `SKILL.md`
- `scripts/`
- `references/`
- `assets/`
- `README*`
- `CHANGELOG*`
- `LICENSE*`
- `NOTICE*`
- `TODO*`
- `.gitignore`
- `.editorconfig`
- `.skillinquisitorignore`
- `pyproject.toml`
- `requirements.txt`
- `requirements-dev.txt`
- `uv.lock`
- `package.json`
- `package-lock.json`
- `pnpm-lock.yaml`
- `yarn.lock`
- `Cargo.toml`
- `Cargo.lock`
- `Gemfile`
- `Gemfile.lock`
- `go.mod`
- `go.sum`
- `.python-version`
- `.tool-versions`

- `D-14A` Nested or duplicate skill manifest
  - Additional `SKILL.md` files below the declared skill root
  - Severity: `MEDIUM`
- `D-14B` Unexpected top-level directory
  - Low-severity for unusual directories, stronger for `node_modules`, `dist`, `build`, `target`, `venv`, `.venv`, `__pycache__`
  - Severity: `LOW` or `MEDIUM`
- `D-14C` Unexpected top-level file
  - Extra root-level text/config file outside the allowlist
  - Severity: `LOW`
- `D-14D` Executable script outside `scripts/`
  - Detect via executable bit, shebang, or both
  - Severity: `MEDIUM`
- `D-14E` Native binary or installer present
  - PE, ELF, Mach-O, shared libraries, installer images, or equivalent
  - Severity: `HIGH`
- `D-14F` Archive or opaque bundle present
  - `.zip`, `.tar`, `.gz`, `.7z`, `.rar`, `.jar`, and similar bundles outside the only explicitly safe context supported by Epic 7:
    - archive located under `assets/`
    - archive name does not begin with `.`
    - archive is not executable
  - Severity: `MEDIUM`
- `D-14G` Suspicious hidden file or directory
  - Unexpected dotfiles or dotdirs not on the allowlist
  - Severity: `LOW` or `MEDIUM`

### D-15 URL Classification

D-15 should scan all original and derived segments so hidden URLs inside comments, code fences, or decoded payloads are still classified.

Context contract for D-15 severity:
- default context is `documentation`
  - severity baseline: `LOW` for unknown external hosts
- `actionable_instruction`
  - markdown or text segment with one of these verbs on the same line or the immediately following line as the URL:
    - `install`
    - `run`
    - `download`
    - `fetch`
    - `curl`
    - `wget`
    - `pip`
    - `npm`
    - `pnpm`
    - `yarn`
    - `cargo`
    - `uv`
    - `execute`
    - `open`
    - `visit`
  - severity baseline: `MEDIUM`
- `executable_snippet`
  - any non-markdown code artifact
  - or a code fence with language in:
    - `bash`
    - `sh`
    - `shell`
    - `python`
    - `py`
    - `javascript`
    - `js`
    - `typescript`
    - `ts`
    - `ruby`
    - `rb`
    - `go`
    - `rust`
  - severity baseline: `MEDIUM`
- `registry_or_dependency`
  - package-manager command, manifest, or config file indicating package source or dependency resolution
  - severity baseline: `HIGH` for suspicious hosts or override behavior

If no stronger signal is present, D-15 should use the `documentation` baseline.

- `D-15A` Allowlisted host
  - `INFO` or optionally suppressed in normal output
- `D-15B` URL shortener host
  - `MEDIUM` in `documentation`
  - `HIGH` in `actionable_instruction`, `executable_snippet`, or `registry_or_dependency`
- `D-15C` IP-literal host
  - IPv4, IPv6, localhost, loopback, or RFC1918/private ranges
  - Severity: `HIGH`
- `D-15D` Obscured IP representation
  - Hex, octal, integer, or mixed-radix host encodings
  - Severity: `HIGH`
- `D-15E` Unknown external host
  - `LOW` in `documentation`
  - `MEDIUM` in `actionable_instruction` or `executable_snippet`
  - `HIGH` in `registry_or_dependency`
- `D-15F` Suspicious URL encoding trick
  - Userinfo host confusion, heavy percent-encoding, double-encoded separators, hex-like path tricks
  - Severity: `MEDIUM` in `documentation`
  - Severity: `HIGH` in `actionable_instruction`, `executable_snippet`, or `registry_or_dependency`
- `D-15G` Non-HTTPS URL where HTTPS is expected
  - Severity: `MEDIUM`
- `D-15H` Punycode or non-ASCII host
  - Severity: `MEDIUM`

URL classification should canonicalize candidates before evaluation:
- normalize `hxxp` to `http`
- normalize `[.]` to `.`
- lowercase hosts
- strip trailing dots
- decode punycode
- percent-decode path once for inspection

### D-20 Package Poisoning and Skill-Name Typosquatting

Package poisoning should combine direct registry-override detection with curated typosquatting checks.

- `D-20A` Python index override
  - `--index-url`, `--extra-index-url`, `--find-links`, pip env vars, pip config, `pyproject.toml` source/index settings
  - Severity: `HIGH` for unknown external hosts
- `D-20B` JavaScript registry override
  - `--registry`, `.npmrc`, scoped registry overrides, tarball/http dependency specs
  - Severity: `HIGH`
- `D-20C` Cargo registry override
  - `[registries]`, `[source.*]`, `replace-with`, `cargo install --registry`
  - Severity: `HIGH`
- `D-20D` Typosquatted package name
  - Curated protected package list with length-aware Damerau-Levenshtein thresholds
  - Severity: `HIGH`
- `D-20E` Dependency-confusion pattern
  - Mixed public/private index posture or ambiguous private-looking package names
  - Severity: `MEDIUM` or `HIGH`
- `D-20F` Skill-name typosquatting
  - Parsed frontmatter `name` suspiciously close to a protected skill-name list
  - Severity: `MEDIUM` or `HIGH`

Typosquatting policy should be length-aware, ecosystem-aware, and allowlisted where needed. It should not try to compare against the entire public package universe.

Exact typosquatting normalization and threshold rules:
- lowercase before comparison
- Python package names:
  - apply PEP 503-style normalization
  - collapse runs of `-`, `_`, and `.` to `-`
- JavaScript package names:
  - if scoped, compare scope and package leaf separately
  - exact scoped match is always safe
  - for scoped typosquatting, require both:
    - normalized scope within threshold of the protected scope
    - normalized leaf within threshold of the protected leaf
  - example protected form: `@anthropic-ai/sdk`
- Rust package names:
  - lowercase
  - treat `-` and `_` as equivalent by normalizing both to `-`
- Skill names:
  - lowercase
  - normalize whitespace and runs of `-` / `_` to a single `-`

Length buckets:
- short name: normalized length `<= 5`
- medium name: normalized length `6..10`
- long name: normalized length `>= 11`

Threshold application:
- short names may differ by at most `1`
- medium names may differ by at most `1`
- long names may differ by at most `2` and must also satisfy `distance / normalized_length <= 0.2`

Additional required guards:
- first character must match after normalization
- token count must match for skill names and unscoped package names
- scoped package comparisons must preserve scope-versus-leaf position
- exact normalized match never triggers

### D-23 File Size and Text-Density Anomalies

D-23 should avoid a naive UTF-8 byte ratio. The better model is corroborated display-density analysis.

- `D-23A` Non-rendered-content inflation
  - Large markdown/text files where hidden regions such as HTML comments dominate byte mass
  - Severity: `MEDIUM` or `HIGH`
- `D-23B` Invisible-unicode mass
  - Large payload contribution from invisible Unicode classes
  - Severity: `HIGH`
- `D-23C` Opaque text blob
  - Large text-like artifact with poor display density plus corroborating suspicious signals such as encoded spans or high-entropy blocks
  - Severity: `LOW` or `MEDIUM`

The implementation should use rendered/display width rather than plain character count, require minimum size thresholds, and limit D-23 to text-like artifacts. Binaries and archives belong primarily under D-14.

Exact D-23 thresholds:
- normalize line endings to LF before measurement
- compute `display_cells` as the sum of rendered cell widths, clamping negative widths to `0`
- do not emit any D-23 finding unless:
  - `byte_size >= 4096`
  - `display_cells >= 500`
- `D-23A`
  - trigger when `non_rendered_bytes / byte_size >= 0.35` and `non_rendered_bytes >= 2048`
  - escalate to `HIGH` when `non_rendered_bytes / byte_size >= 0.60`
- `D-23B`
  - trigger when `invisible_bytes >= 1024` and `invisible_bytes / byte_size >= 0.20`
- `D-23C`
  - trigger when `bytes_per_display_cell >= 3.0` and at least one corroborating signal is present:
    - any Epic 4 encoding finding on the same artifact
    - `non_rendered_bytes >= 2048`

## Config Design

Epics 6 and 7 need modest but important config expansion.

### Frontmatter Policy

The default frontmatter allowlist should be:
- `name`
- `description`
- `disable-model-invocation`

`disable-model-invocation` is included because the project research already treats auto-invocation posture as relevant skill metadata even though it is not central to current BRD examples. The field should be treated as an allowed optional extension, not as a required field.

Recommended config schema:

```yaml
frontmatter_policy:
  allowed_fields:
    - name
    - description
    - disable-model-invocation
  field_types:
    name: str
    description: str
    disable-model-invocation: bool
  description_max_length: 500
```

The implementation should allow extension for future ecosystem-specific fields without forcing code changes for every metadata addition.

### URL Policy

The flat `trusted_urls` config should remain supported, but the durable direction is a structured URL policy that can represent:
- allowed hosts
- allowed domain suffixes
- allowed schemes
- known shortener hosts
- registry hosts by ecosystem
- custom-index allowlist hosts
- whether allowlisted URLs are reported at all

Recommended config schema:

trusted_urls:

url_policy:
  allow_hosts:

    - pypi.org
    - files.pythonhosted.org

    - https

## Testing Strategy

The regression harness should remain the source of truth.

### Fixture Taxonomy

Add `tests/fixtures/deterministic/injection/` for:
- exact injection phrases
- safe quoted or documentary examples
- suppression directives
- frontmatter-description attacks
- malformed frontmatter
- combo fixtures where suppression coexists with Epic 5 behavior

Add `tests/fixtures/deterministic/structural/` for:
- missing or malformed skill structure
- executables, binaries, and archives
- safe root-level files and directories
- allowlisted versus unknown URLs
- registry override cases by ecosystem
- package near-miss safe and positive cases
- skill-name near-miss safe and positive cases
- D-23 anomaly positives and multilingual safe negatives

### Scope Policy

New malicious deterministic fixtures should usually use scoped matching:
- `layers: ["deterministic"]`
- `checks: [...]`

This keeps the deterministic suite stable once Epic 9 and Epic 10 start adding valid findings on the same artifacts.

### Fixture Schema Additions

`expected.yaml` should gain these optional keys:
- `config_override`
  - deep-merged against `ScanConfig` defaults for this fixture only
- `action_flags_contains`
  - list of selectors plus required flags
- `details_contains`
  - list of selectors plus required detail snippets

Each selector should target exactly one expected finding using:
- `rule_id`
- `file_path`
- `start_line`

The assertion should match exactly one actual finding after normal fixture matching has already established the expected finding set.

Example:

```yaml
schema_version: 1
verdict: MEDIUM RISK
match_mode: exact
scope:
  layers: [deterministic]
  checks: [D-12A]
config_override:
  url_policy:
    allow_hosts: [example.com]
findings:
  - rule_id: D-12A
    layer: deterministic
    category: suppression
    severity: medium
    message: Concealment or non-disclosure directive detected
    location:
      file_path: SKILL.md
      start_line: 4
      end_line: 4
action_flags_contains:
  - selector:
      rule_id: D-12A
      file_path: SKILL.md
      start_line: 4
    flags: [SUPPRESSION_PRESENT, SUPPRESS_DISCLOSURE]
details_contains:
  - selector:
      rule_id: D-12A
      file_path: SKILL.md
      start_line: 4
    values:
      amplifier_eligible: true
```

### Test-Support Coverage

Add direct unit tests for:
- frontmatter extraction and span mapping
- skill-scope rule execution
- URL canonicalization and allowlist behavior
- typosquatting threshold behavior
- D-12 metadata emission

## Risks and Guardrails

The main risks are architectural, not just regex quality.

- If frontmatter is not extracted first, D-13 and skill-name typosquatting will be brittle.
- If binary and executable metadata are still discarded in `input.py`, D-14 and D-23 will miss exactly the artifacts they are meant to catch.
- If `skill` scope is skipped, structure validation will leak into awkward artifact hacks.
- If deterministic injection rules become too broad, Epic 9 loses its reason to exist and false positives will climb.
- If the fixture harness cannot assert metadata and config behavior, suppression amplifiers and URL allowlists will be under-specified.

The robust implementation posture is:
- precise deterministic rules
- richer evidence
- scoped fixtures
- deliberate handoff to later ML/LLM/scoring layers for the ambiguous cases

## Overlap and Dedup Rules

To avoid noisy duplicate findings, Epics 6 and 7 should follow these rules:

- `D-13E` is contextual and frontmatter-specific. It should fire only on the `FRONTMATTER_DESCRIPTION` segment or frontmatter field span, not duplicate generic `D-11` or `D-12` findings on the same raw span unless the generic finding lands on a materially different span.
- generic `D-11` and `D-12` findings that arise from `FRONTMATTER_DESCRIPTION` should suppress identical `D-11` or `D-12` findings on the `ORIGINAL` segment when:
  - `rule_id` matches
  - `message` matches
  - the raw span on `ORIGINAL` lies entirely within `frontmatter_location`
- URL findings should dedupe by canonical URL plus raw file path plus raw source line. If the same URL appears in both an original segment and a derived segment representing the same raw span, emit only one finding.
- Registry override findings and URL findings may coexist when they represent different semantics.
- Structure findings should not be re-emitted per artifact if they are fundamentally skill-level conditions.

## Recommended Implementation Sequence

The approved implementation order is:

1. substrate and fixture-harness upgrades
2. Epic 6 deterministic rules and fixtures
3. Epic 7 deterministic rules and fixtures
4. docs and requirements sync

This order keeps the risky plumbing work in front of the rule work and minimizes rework across both epics.
